# Pine Design System Audit Prompt

Review Pine Design System as a mature design-system documentation site.

Do not just add more sidebar sections. Improve the quality and usefulness of the existing documentation so it feels comparable to systems like Carbon, Polaris, Material, and GOV.UK.

For each existing section, strengthen:

1. Decision guidance
Explain when to use each component or pattern, when not to use it, and what to use instead.

2. Real product examples
Show components in realistic page contexts, not only isolated Storybook examples.

3. Anatomy and behavior
For each component, document its parts, variants, states, responsive behavior, keyboard behavior, and accessibility expectations.

4. Do and don't examples
Add visual and written examples of correct and incorrect usage.

5. Content guidance
Include label, helper text, error message, button text, and empty-state writing guidance.

6. Accessibility details
Go beyond accessible claims. Include focus behavior, ARIA expectations, keyboard interaction, contrast requirements, and testing notes.

7. Token usage
Explain not only token names and values, but why and when to use each token. Show semantic token examples.

8. Pattern-level guidance
Show how components combine into forms, navigation, cards, settings screens, data displays, and full-page layouts.

9. Governance and maturity
Add component status, versioning, changelog, contribution guidance, and design/code parity expectations.

10. Editorial quality
Make the documentation feel intentional, opinionated, and teachable. Each page should help a team make better product decisions, not just inspect available UI pieces.

## Sharper Framing

Pine may already have the information architecture of a design system. What might still separate it from the bigger references is likely depth of guidance, product examples, governance, and opinionated usage rules.
